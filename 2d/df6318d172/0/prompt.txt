# jq/grep Tool Security Audit — Findings

Verified against `jq-1.7.1` (prod runs 1.7; lexer/parser behavior identical for comments) by running the **exact tool argv** (`jq -- '&lt;query&gt;' &lt;file&gt;`).

## FINDING 1 — CRITICAL: `_MODULE_DIRECTIVE` regex bypassed by a `#` comment between the keyword and the string

**Severity:** Critical (arbitrary cross-workspace file read — the exact primitive the regex was added to stop).

**Root cause:** The regex `(?&lt;![\w"])(?:import|include)\s*"` only allows **whitespace** (`\s*`) between `import`/`include` and the opening quote. jq's lexer treats a `#` line comment as inter-token whitespace, so a comment between the keyword and the module string is syntactically valid to jq but does **not** match `\s*"`.

**Exact input that triggers it** (the precise argv the tool builds for `raw=False`):
```
jq -- 'include #
"conf" {search:"/abs/path/to/secretmod"}; secret_api_key' data.json
```

**Verified actual behavior:**
- Python: `_MODULE_DIRECTIVE.search(query)` → `False` (NOT blocked).
- jq loads `secretmod/conf.jq` from the attacker-controlled `{search:...}` path and runs `secret_api_key`, printing `sk-PROD-1234567890` (the module body content). The `{search:"..."}` metadata lets the attacker point the loader at any directory on disk.
- Multiple/blank comment lines (`include #a\n#b\n "m" ...`) work identically.
- `include` is the reliable vector; `import "m" as $m {search}` returned "module not found" in 1.7.1 due to stricter import resolution, but `include` alone is a full read primitive, so the hole is real regardless.

Constraints (don't reduce severity meaningfully): the target file must be parseable as jq (so it reads `.jq`-style/jq-valid files, and leaks their definitions). Any attacker-staged or jq-syntax-compatible file in the workspace/user dir is exfiltratable; the search path is arbitrary.

**The fix (robust replacement — strip strings + `#` comments, THEN match the keyword token), verified against all bypass and false-positive cases:**
```python
_JQ_STRING = re.compile(r'"(?:\\.|[^"\\])*"')   # jq double-quoted string (handles escapes)
_JQ_COMMENT = re.compile(r"#[^\n]*")            # jq # line comment to EOL
_MODULE_KEYWORD = re.compile(r"(?&lt;![\w.])(?:import|include)\b")

def _has_module_directive(query: str) -&gt; bool:
    # Neutralize string literals first (so " import " in data isn't a keyword),
    # then strip # comments (so they can't smuggle whitespace), then look for the
    # bare keyword token. Order matters: strings before comments, since a # inside
    # a string is not a comment.
    stripped = _JQ_COMMENT.sub(" ", _JQ_STRING.sub('""', query))
    return bool(_MODULE_KEYWORD.search(stripped))
```
This blocks `plain`, `comment-bypass (import &amp; include)`, `tab/multi-space`, and `multi-comment-line` forms, and does NOT false-positive on `.["import"]`, `select(.subject|contains(" import "))`, `.import`, or a comment that merely mentions "import".

**pytest test that catches it** (add to `test_jq_grep_tools.py`; it FAILS today):

### 2. Is unprivileged userns available (would bwrap work)? — **Probably not; drop bwrap.**

Three independent blockers, each fatal on its own:

1. **Docker default seccomp.** Docker's default profile allows `unshare`/`clone` with `CLONE_NEWUSER` **only if the container has `CAP_SYS_ADMIN`** (the profile gates namespace-creation syscalls on that cap). Prod gaia-backend has **no `CAP_SYS_ADMIN`** and no custom seccomp → `clone(CLONE_NEWUSER)` / `unshare(CLONE_NEWUSER)` is **blocked by seccomp (EPERM)**. bwrap's first act is exactly this. This alone kills bwrap in prod.

2. **Ubuntu AppArmor userns restriction.** Ubuntu 23.10+ and backports restrict unprivileged userns via `kernel.apparmor_restrict_unprivileged_userns=1`. 22.04 is less likely to enforce this, but the docker-default AppArmor profile is still in force and the host sysctl `kernel.unprivileged_userns_clone` may be 0 on a hardened Hetzner image. Unverified, but another possible block.

3. **No `no-new-privileges` doesn't help** — bwrap doesn't need setuid (it's userns-based), but that's moot given #1.

**Conclusion:** With the default seccomp profile and no `SYS_ADMIN`, `bwrap` cannot create a user namespace. **bwrap is not deployable here.** The proposal's own fallback ordering should be inverted.

*Must be verified on the real host* (see §6), but the seccomp gate is essentially certain given the compose config.

---

### 3. Is Landlock + seccomp a viable fallback? — **Yes; this is the right primary.**

- **Landlock does NOT need any capability or namespace.** It's an unprivileged, task-self-sandboxing LSM: any thread can call `landlock_create_ruleset` + `landlock_add_rule` + `landlock_restrict_self` to drop its own filesystem access, and it's inherited across `execve`. This sidesteps every blocker in §2. It is **allowed by Docker's default seccomp** (the landlock syscalls aren't in the default denylist).
- **Kernel:** ABI v1 (path-based FS allow-listing) is ≥5.13 → present on 22.04. That's all we need: allow-list read-only access to (a) the single resolved target file, (b) jq's binary `/usr/bin/jq` + its shared libs (`/lib`, `/usr/lib` read+execute), (c) `/dev/null`. Everything else → denied at the kernel. **Landlock v1 cannot restrict network** — that's what seccomp is for.
- **seccomp net-block:** a tiny BPF filter that returns `EACCES`/`EPERM` for `socket`, `socketcall`, `connect`, `bind`. jq has no legitimate socket use, so this is safe. Apply it in the same `preexec_fn`. Applying a *second, stricter* seccomp filter on top of Docker's is always allowed (filters stack; you can only add restrictions).
- **Python application, pre-exec:** the existing `_filter.py:55-77 _apply_child_limits()` is **already a `preexec_fn`** running post-fork/pre-exec. Landlock + seccomp go **right here** via `ctypes` raw `syscall()` (Landlock has no glibc wrapper) and a hand-assembled seccomp BPF (or the `seccomp`/`pyseccomp` lib). Effort: ~150–250 lines of ctypes. Robustness: **high** — kernel-enforced, no privilege needed, fail-closed if the syscall errors.
- One caveat: `preexec_fn` runs in the forked child between fork and exec; it must be async-signal-safe-ish (no Python allocation storms). Landlock/seccomp are a handful of syscalls — fine, same risk profile as the `setrlimit` calls already there.

### 4. E2B fallback path — concrete routing

The target file already lives in the **shared JuiceFS namespace**, so no copy is needed:

- `_filter.py:126` calls `resolve_user_file(user_id, rel)` → `juicefs.py:248` returns the host `Path` under `/mnt/jfs/...`. The E2B sandbox mounts the **same** JuiceFS volume (`docker-compose.yml:73-78`: "the API host and the E2B sandboxes must share one JuiceFS namespace"). So the file the model named is visible inside the sandbox at its workspace path.
- `bash_tool.py:209` shows the pattern: `async with acquire_sandbox(user_id) as sbx:` yields a sandbox exposing `sbx.commands.run(cmd, timeout=…)` and `sbx.files.read/write` (`lifecycle.py`). `acquire_sandbox` is an async contextmanager (`app/services/sandbox/__init__.py`, imported at `bash_tool.py:43-45`).
- **jq is already in the E2B template** (`scripts/build_e2b_template.py:130` `.apt_install(... "jq" ...)`), so no template change is needed for the fallback.
- **Routing:** when host-confinement is unavailable, `run_file_filter` acquires a sandbox and runs `jq &lt;argv&gt; &lt;workspace-relative-path&gt;` via `sbx.commands.run`, applying the same timeout/output cap on the returned stdout. The sandbox is already OS-isolated (E2B microVM, no-sudo, `hidepid`, non-dumpable JuiceFS launcher — see `apps/api/CLAUDE.md` sandbox hardening notes), so no extra confinement is layered.
- Cost tradeoff: sandbox acquire is heavier than an in-process exec (that's the whole reason jq was pulled host-side per `__init__.py:4-5`), so E2B should be **fail-closed fallback only**, not the steady state.

---

### 5. Concrete proposal

**Mechanism: Landlock (FS) + seccomp (net) in the existing `preexec_fn`, gated by a startup canary; E2B on canary-fail. Drop bwrap entirely.**

**Dockerfile changes:** essentially none for the primary path — jq is already installed (`Dockerfile:48`), Landlock/seccomp are kernel features needing no package. If using `pyseccomp` for the BPF, add it to the backend dependency group (avoids hand-rolling BPF). No new binaries, no `bwrap` install.

**`preexec_fn` additions (in `_filter.py:_apply_child_limits`):**
- Landlock v1: `landlock_create_ruleset(&amp;attr, size, 0)`; add read-only-file rules (`LANDLOCK_ACCESS_FS_READ_FILE`) for the resolved target fd, and read+execute (`READ_FILE|EXECUTE`) rules for `/usr/bin/jq`, `/usr/lib`, `/lib`, `/lib64`; then `prctl(PR_SET_NO_NEW_PRIVS,1,0,0,0)` (required before restrict) + `landlock_restrict_self(fd,0)`.
- seccomp: BPF returning `SECCOMP_RET_ERRNO(EPERM)` for `socket`, `connect`, `bind`, `socketcall`, then `SECCOMP_RET_ALLOW`. (`PR_SET_NO_NEW_PRIVS` above satisfies seccomp's precondition too.)
- Keep the existing rlimits (`RLIMIT_CPU`, `RLIMIT_FSIZE=0`, `RLIMIT_AS`).

**Startup capability-probe + confinement canary (must pass or host-jq stays OFF):**
- At startup (lazy-provider registration), fork a child that (a) applies Landlock+seccomp, then (b) attempts two things that MUST fail: read a file outside the allow-list (e.g. `/etc/passwd` → expect EACCES) and open a TCP socket (expect EPERM); and one that MUST succeed: read the jq binary. If the negative tests don't deny and the positive test doesn't allow, **set a module flag `HOST_JQ_CONFINED=False`**.
- `run_file_filter` checks that flag: `True` → in-process confined exec (current path + Landlock/seccomp); `False` → route to E2B (`acquire_sandbox`). **Never run unconfined host-jq.** This is the fail-closed guarantee.

**`_filter.py` integration sketch (minimal):**
```
# module-level, set once by the startup canary
HOST_JQ_CONFINED: bool = _run_confinement_canary()

async def run_file_filter(...):
    ...
    target = await resolve_user_file(user_id, rel)
    if HOST_JQ_CONFINED:
        return await _run(program, args, str(target), ...)   # preexec now adds Landlock+seccomp
    return await _run_in_sandbox(user_id, binary, args, rel, ...)  # E2B fallback
```
The `preexec_fn` gains a `_apply_confinement(target_path)` call after the existing rlimits. Because Landlock needs the target path, `_apply_child_limits` becomes a closure over `target` (or `functools.partial`).

### 6. Unknowns requiring verification on real infra

1. **Exact prod host kernel** — confirm `uname -r` on the Hetzner Swarm node (drives Landlock ABI: need ≥5.13 for v1). 22.04 default is 5.15; verify no older HWE/custom kernel.
2. **`kernel.apparmor_restrict_unprivileged_userns` / `kernel.unprivileged_userns_clone` sysctls** on the host — confirms bwrap is dead (expected) *and* that these don't affect Landlock (they don't, but verify the container can call the landlock syscalls under docker-default AppArmor).
3. **Landlock not blocked by docker-default seccomp/AppArmor** — run the canary inside an actual prod-equivalent container (`landlock_create_ruleset` returns a valid fd, not `ENOSYS`/`EPERM`).
4. **seccomp filter stacking** — confirm adding a second filter succeeds under the default profile (it should; only additive restrictions).
5. **`preexec_fn` safety** — verify the ctypes syscall sequence in the forked child doesn't deadlock/misbehave under the async event loop (the codebase already flags this at `_filter.py:152`).
6. **jq's actual dynamic-lib set** at runtime (`ldd $(which jq)` in the prod image) so the Landlock read+execute allow-list is complete — an incomplete list makes jq fail to start under confinement, which the canary must catch.

**Recommendation:** implement Landlock+seccomp as primary, E2B as fail-closed fallback, delete bwrap from the design.
