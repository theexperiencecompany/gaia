‚Źļ The verbosity comes from a fundamental Docker Swarm limitation: secrets are
  mounted as files at /run/secrets/<name>, but your app expects environment 
  variables. There's no native way to bridge that ‚ÄĒ Swarm has no equivalent of
  Kubernetes' secretKeyRef.

  Your options:

  Option A ‚ÄĒ Entrypoint script in the image (cleanest compose file)

  Add to each Dockerfile:
  COPY scripts/docker-entrypoint.sh /entrypoint.sh
  ENTRYPOINT ["/entrypoint.sh"]

  scripts/docker-entrypoint.sh:
  #!/bin/sh
  [ -f /run/secrets/gaia_infisical_token ]                          && export
  INFISICAL_TOKEN=$(cat /run/secrets/gaia_infisical_token)
  [ -f /run/secrets/gaia_infisical_machine_identity_client_id ]     && export
  INFISICAL_MACHINE_IDENTITY_CLIENT_ID=$(cat
  /run/secrets/gaia_infisical_machine_identity_client_id)
  [ -f /run/secrets/gaia_infisical_machine_identity_client_secret ] && export
  INFISICAL_MACHINE_IDENTITY_CLIENT_SECRET=$(cat
  /run/secrets/gaia_infisical_machine_identity_client_secret)
  [ -f /run/secrets/gaia_infisical_project_id ]                     && export
  INFISICAL_PROJECT_ID=$(cat /run/secrets/gaia_infisical_project_id)
  [ -f /run/secrets/gaia_metrics_token ]                            && export
  METRICS_TOKEN=$(cat /run/secrets/gaia_metrics_token)
  exec "$@"

  Then in docker-compose.prod.yml the command is just the normal one:
  command: python -m uvicorn app.main:app --host 0.0.0.0 --port 80 ...

  Tradeoff: the mapping is now invisible in the compose file and requires an
  image rebuild to change.

  Option B ‚ÄĒ App reads /run/secrets/ directly

  In your Python config/settings, read the file if the env var isn't set:

  def _read_secret(env_var: str, secret_name: str) -> str | None:
      if val := os.getenv(env_var):
          return val
      path = Path(f"/run/secrets/{secret_name}")
      return path.read_text().strip() if path.exists() else None

  INFISICAL_TOKEN = _read_secret("INFISICAL_TOKEN", "gaia_infisical_token")

  Tradeoff: cleanest at runtime (no wrapper at all), but touches app code.

  Reality check

  The sh -c pattern in the compose file is the standard Docker Swarm idiom ‚ÄĒ
  you'll see it everywhere. It's verbose but it's all in one place and requires
  no image or code changes. For 5 bootstrap secrets that almost never change,
  it's not unreasonable.

  The entrypoint approach is worth it if you have many services all repeating
  the same block (which you do ‚ÄĒ backend, worker, 4 bots). One shared
  docker-entrypoint.sh in the repo, all Dockerfiles reference it, compose file
  stays clean.

‚Źļ Clean. Now addressing your question about the full CI/CD pipeline and Swarm
  capabilities:

  ---
  What you have now

  Auto-healing ‚ÄĒ yes, fully set up:
  - restart_policy: condition: any on every service ‚Üí Swarm restarts a container
   if it exits for any reason
  - healthcheck on the API ‚Üí Swarm marks the task unhealthy and replaces it if
  it fails retries: 5 times
  - failure_action: rollback on update_config ‚Üí if a rolling update produces an
  unhealthy container, Swarm auto-rolls back to the previous spec

  Zero-downtime deploys ‚ÄĒ yes, configured:
  - order: start-first ‚Üí Swarm starts the new container and waits for it to pass
   its healthcheck before stopping the old one. At no point is the service down.
  - monitor: 45s ‚Üí gives the new task 45 seconds to stabilize after starting
  before declaring the update successful

  Rollback ‚ÄĒ yes, two modes:
  - Manual: GitHub Actions ‚Üí Deploy Swarm Production ‚Üí action=rollback ‚Üí
  rollback_mode=last (uses Docker's built-in previous spec) or
  rollback_mode=digest (redeploys a specific image SHA)
  - Automatic: failure_action: rollback kicks in during CI deploy if health
  fails

  ---
  Gaps worth knowing about

  1. No health wait after deploy ‚ÄĒ the CI job exits as soon as docker stack 
  deploy returns. That command just submits the desired state to Swarm; the
  actual convergence (pulling image, starting container, passing healthcheck)
  happens asynchronously. If the deploy silently fails after CI exits, nobody
  knows. You could add a convergence poll:
  docker --context prod service ps gaia-prod_gaia-backend \
    --filter desired-state=running --format '{{.CurrentState}}' | grep -q
  "Running"
  But given failure_action: rollback is set, Swarm handles this itself ‚ÄĒ it's
  more of a CI visibility gap than a reliability gap.

  2. No resource limits on bots ‚ÄĒ the four bot services have no
  resources.limits.memory. If a bot leaks memory it can starve infra services.
  Easy fix: add memory: 256m or similar to their deploy.resources.

  3. Infra services have no healthchecks in the stack file ‚ÄĒ postgres, redis,
  chromadb, rabbitmq have healthcheck in docker-compose.yml (dev) but if those
  blocks aren't in docker-compose.prod.yml, depends_on conditions are ignored by
   Swarm anyway (Swarm ignores depends_on entirely). Not a blocker since the
  API's own healthcheck retries cover transient infra startup.

  4. No deploy notification on rollback from CI ‚ÄĒ the rollback job Discord
  notification exists, but automatic Swarm rollbacks (triggered by
  failure_action: rollback) are invisible. No alert fires if Swarm silently
  rolls back during a deploy. This is the main observability gap.

  Want me to add resource limits to the bots, or set up a convergence check in
  the deploy step?
