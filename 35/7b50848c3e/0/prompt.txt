# jq/grep Tool Security Audit — Findings

Verified against `jq-1.7.1` (prod runs 1.7; lexer/parser behavior identical for comments) by running the **exact tool argv** (`jq -- '&lt;query&gt;' &lt;file&gt;`).

## FINDING 1 — CRITICAL: `_MODULE_DIRECTIVE` regex bypassed by a `#` comment between the keyword and the string

**Severity:** Critical (arbitrary cross-workspace file read — the exact primitive the regex was added to stop).

**Root cause:** The regex `(?&lt;![\w"])(?:import|include)\s*"` only allows **whitespace** (`\s*`) between `import`/`include` and the opening quote. jq's lexer treats a `#` line comment as inter-token whitespace, so a comment between the keyword and the module string is syntactically valid to jq but does **not** match `\s*"`.

**Exact input that triggers it** (the precise argv the tool builds for `raw=False`):
```
jq -- 'include #
"conf" {search:"/abs/path/to/secretmod"}; secret_api_key' data.json
```

**Verified actual behavior:**
- Python: `_MODULE_DIRECTIVE.search(query)` → `False` (NOT blocked).
- jq loads `secretmod/conf.jq` from the attacker-controlled `{search:...}` path and runs `secret_api_key`, printing `sk-PROD-1234567890` (the module body content). The `{search:"..."}` metadata lets the attacker point the loader at any directory on disk.
- Multiple/blank comment lines (`include #a\n#b\n "m" ...`) work identically.
- `include` is the reliable vector; `import "m" as $m {search}` returned "module not found" in 1.7.1 due to stricter import resolution, but `include` alone is a full read primitive, so the hole is real regardless.

Constraints (don't reduce severity meaningfully): the target file must be parseable as jq (so it reads `.jq`-style/jq-valid files, and leaks their definitions). Any attacker-staged or jq-syntax-compatible file in the workspace/user dir is exfiltratable; the search path is arbitrary.

**The fix (robust replacement — strip strings + `#` comments, THEN match the keyword token), verified against all bypass and false-positive cases:**
```python
_JQ_STRING = re.compile(r'"(?:\\.|[^"\\])*"')   # jq double-quoted string (handles escapes)
_JQ_COMMENT = re.compile(r"#[^\n]*")            # jq # line comment to EOL
_MODULE_KEYWORD = re.compile(r"(?&lt;![\w.])(?:import|include)\b")

def _has_module_directive(query: str) -&gt; bool:
    # Neutralize string literals first (so " import " in data isn't a keyword),
    # then strip # comments (so they can't smuggle whitespace), then look for the
    # bare keyword token. Order matters: strings before comments, since a # inside
    # a string is not a comment.
    stripped = _JQ_COMMENT.sub(" ", _JQ_STRING.sub('""', query))
    return bool(_MODULE_KEYWORD.search(stripped))
```
This blocks `plain`, `comment-bypass (import &amp; include)`, `tab/multi-space`, and `multi-comment-line` forms, and does NOT false-positive on `.["import"]`, `select(.subject|contains(" import "))`, `.import`, or a comment that merely mentions "import".

**pytest test that catches it** (add to `test_jq_grep_tools.py`; it FAILS today):
```python
@needs_jq
async def test_jq_tool_blocks_include_comment_bypass(tmp_path: Path) -&gt; None:
    secret_dir = tmp_path / "mod"; secret_dir.mkdir()
    (secret_dir / "conf.jq").write_text('def k: "TOPSECRET";')
    target = tmp_path / "mine.json"; target.write_text("{}")
    query = f'include #c\n"conf" {{search:"{secret_dir}"}}; k'
    with _mock_resolve(target):
        out = await jq.ainvoke({"query": query, "path": "mine.json", "raw": True}, config=CONFIG)
    assert "TOPSECRET" not in out      # FAILS now: module IS loaded
    assert "not allowed" in out
```

## FINDING 2 — False positives in the CURRENT regex: none of the tested legit queries are wrongly rejected

**Severity:** None (defense verified correct here).
- `.["import"]` → regex `False` (not blocked). Already covered by `test_jq_tool_allows_field_named_import`.
- `select(.subject | contains(" import "))` → regex `True` because `import` is followed by `"`... but verified the lookbehind/`"`-requirement means it matches the `import "` **inside the string**. So this legit query **is wrongly rejected** by the current regex. Confirmed: `REGEX_MATCH=True`.

So Finding 2 IS a real (low-severity) false positive: a query whose string contains the substring `import "` (e.g. `contains("... import \"...")`, or any `.subject` filter mentioning an import keyword immediately before a quote) is blocked even though it loads no module. The proposed Finding-1 fix (strip strings first) eliminates this.

**pytest test** (FAILS today, PASSES after the fix):
```python
@needs_jq
async def test_jq_tool_allows_keyword_inside_string(tmp_path: Path) -&gt; None:
    f = tmp_path / "d.json"; f.write_text('{"subject":"please import \\"x\\""}')
    with _mock_resolve(f):
        out = await jq.ainvoke(
            {"query": 'select(.subject | contains("import \\""))|.subject',
             "path": "d.json", "raw": True}, config=CONFIG)
    assert "not allowed" not in out      # FAILS now: string content false-positives
```

## FINDING 3 — Alternate file-read / env-leak vectors: all confirmed BLOCKED (defenses hold)

Ran each against the exact safe env (`PATH=/usr/bin:/bin`, `LC_ALL=C`) and DEVNULL stdin:
- `env` / `$ENV` → returns **only** `{"PATH","LC_ALL"}`. Secret-free env defense holds (no DB creds/Infisical leak).
- `input` → errors (`break`); `[inputs]` → `[]`. stdin DEVNULL defense holds.
- `$__loc__` → `{"file":"&lt;top-level&gt;","line":1}` — no path leak.
- `builtins`/`modulemeta`/`getpath` → no file read without a valid resolvable module path; `modulemeta "m"` → "module not found". No leak.

No action needed; these are the working halves of the hardening. They're worth keeping the existing `test_security_env_is_not_inherited` for.

## FINDING 4 — Keyword case/encoding tricks: NOT a bypass

`Include`, `IMPORT`, `Import` → jq raises a **syntax error** (jq keywords are case-sensitive; capitalized forms aren't recognized as module directives at all). So case tricks can't load a module — they just fail. No finding.

## FINDING 5 — `--` neutralization of `-n`/`--stream`: CONFIRMED WORKING

`jq -- '-n' file` → jq treats `-n` as the **filter program** (`n/0 is not defined`), not the `-n` flag. Same for `--stream`. The `--` guard works as documented. No finding. (A regression test asserting `jq.ainvoke({"query":"-n",...})` returns a `jq error:` rather than null-input behavior would lock this in.)

## FINDING 6 — jq docstring example misfires on a JSON-array file (usability, not security)

**Severity:** Low (correctness/UX). The docstring says it filters "JSON or JSONL" and shows `select(.from | contains("github")) | .subject`. Verified: on a **JSONL** file this works per-line; on a top-level **JSON array** file it errors `Cannot index array with string "from"`. The LLM, steered by the docstring, will apply the JSONL-style query to an array file and get an error instead of results. The docstring's own array example (`[.[] | .id]`) is correct; the issue is the mixed guidance. Recommend clarifying that per-record filters need `.[]` for array files. Not a vulnerability.

## FINDING 7 — grep on a binary file leaks the absolute workspace path (minor info-leak / UX)

**Severity:** Low. Verified: matching a binary file yields `Binary file &lt;abs-path&gt; matches` (BSD here; GNU grep on prod emits the same `Binary file &lt;path&gt; matches` message). Two issues: (a) it exposes the internal resolved absolute path (e.g. `/mnt/jfs/...`) to the model/output, and (b) it returns no useful content. If matching binary content is desired, the tool would need `-a`; if path-hiding matters, strip the resolved prefix from grep's stderr/stdout. Other grep argv cases verified clean: `-e '--'` keeps `--` as a literal pattern (exit 1, no match in test data, no flag confusion — covered by existing `test_grep_tool_pattern_dash_is_literal_not_flag`); empty pattern `-e ''` matches every line (exit 0); invalid regex `[` → exit 2 surfaced as `grep error:` (covered).
