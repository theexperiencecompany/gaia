  Problem: One monolithic quality-checks job runs 7 things in parallel inside a
  single Dagger call. Everything waits for everything, lint failures are hidden

  Solution: Split into focused jobs with Nx affected detection:

  detect (ubuntu-latest, ~30s)         ← uses nx show projects --affected
    │                                     (same pattern as build.yml)
    │
    ├─► static-checks-backend   if: has_backend
    ├─► static-checks-frontend   if: has_frontend
    ├─► dead-code                if: either
    ├─► release-validation       always
    ├─► trivy-scan               always (non-blocking)
    │
    ├─► build                    if: has_frontend (backend build is noop)
    │
    ├─► test-backend             if: has_backend (with 5 service containers)
    ├─► test-frontend            if: has_frontend
    │

  The core tension: Nx affected needs .git, Dagger excludes .git

  Your _IGNORE list strips .git from the Dagger context (rightfully — it's huge
  and slows down filesync). But without git history, nx affected can't work
  inside the container. So nx run-many runs everything regardless of what
  changed.

  build.yml already solved this — it runs nx show projects --affected on the 
  runner (outside Dagger), then uses the results to conditionally skip Docker
  builds. Same pattern here.

  How to make them play together

  The plan currently proposes coarse boolean flags (has_backend/has_frontend).
  That's wasteful. If you change one bot, you still lint all 6 backend projects.

  Better: pass the actual affected project list into Dagger.

  detect job (ubuntu-latest, ~30s):
    → nx show projects --affected
    → outputs: backend_projects="api,bot-discord"  frontend_projects=""

  static-checks-backend:
    → dagger call static-checks-backend --projects="api,bot-discord"
    → inside Dagger: nx run-many -t lint -p api,bot-discord

  Nx already resolves the dependency graph for you. If shared-python changes, nx
   show projects --affected returns
  api,voice-agent,shared-python,bot-discord,... because they all depend on it.
  You just forward that list.

  The Dagger functions get an optional projects param:
  - CI passes it → runs only affected projects
  - Local use omits it → dagger call lint runs everything (same as today)

  The test split is structural, not cosmetic

  Backend tests need 5 service containers (Postgres, Redis, Mongo, Chroma,
  RabbitMQ). Frontend tests need nothing. That's why they must be separate jobs
  — you don't want to spin up databases when only CSS changed.

  Your current naming is confusing:
  - test() runs JS tests AND Python tests without services
  - integration_test() runs Python tests WITH services
  - service_test() runs only service-marked tests WITH services

  New naming:
  - test_backend() — all Python tests with live services. Period.
  - test_frontend() — JS/TS tests via Nx. Period.
  - test() — convenience, runs both (for local use)

  The old test() running Python tests without services was a half-measure. You
  already have USE_REAL_SERVICES=1 wired up. Just always run with real services
  — that's what gets deployed.

  The Dagger overhead question

  Each job rebuilds ci_env() (installs pnpm, uv, deps). On Blacksmith, each job
  gets a fresh VM, so Dagger cache volumes aren't shared. That's ~60-90s per
  job. With 7 jobs, that's ~10 min of setup.

  But the monolith already takes 30-40 minutes wall clock. With the split, your
  longest path is detect (30s) → static-checks-backend (3 min) → test-backend 
  (15 min) = ~19 min. The parallelism more than compensates.

  What I'd actually ship

  detect                          (ubuntu-latest, 30s)
    │
    ├─► static-checks-backend    (lint + mypy, skipped if no backend projects)
    ├─► static-checks-frontend   (lint + tsc, skipped if no frontend projects)

    │
    └─► quality-gate             (summary job for branch protection)

  Each Dagger function accepts --projects so Nx only runs on affected projects
  inside the container.

  quality_checks() stays for dagger call quality-checks locally.

⏺ The 19 min comes from making tests wait for static checks. That's actually a
  regression from the current monolith which runs everything in parallel.

  Drop the sequential gates. There's no real reason test-backend needs to wait
  for lint to pass. If lint fails, you see it in 2 minutes regardless. Meanwhile
   tests are already running.

  detect (30s)
    │
    ├─► static-checks-backend    (~3 min)
    ├─► static-checks-frontend   (~3 min)
    ├─► dead-code                (~2 min)
    ├─► release-validation       (~30s)
    ├─► trivy-scan               (~2 min)
    ├─► build                    (~5 min)
    ├─► test-frontend            (~3 min)
    ├─► test-backend             (~15 min)  ← the bottleneck
    │
    └─► quality-gate (waits for all)

  Critical path: 30s + 15 min = ~16 min. Better, but test-backend is still the
  bottleneck.

  What's actually slow in test-backend

  Roughly:
  - ci_env() rebuild (apt-get, pnpm install, uv sync): ~60-90s
  - Service containers starting (5 of them): ~30s
  - Actual pytest execution with coverage: ~12-13 min

  Three things that would cut this significantly

  1. pytest-xdist for parallel test execution

  You're running pytest on a 4-vCPU runner but executing tests sequentially.
  Adding -n auto distributes across cores.

  "uv", "run", "pytest", "-n", "auto", "-m", "not composio", ...

  Typical speedup: 2-3x. So 12 min → 4-6 min.

  This is the single biggest win. Everything else is optimizing seconds.

  2. Pre-built CI base image

  ci_env() runs apt-get update, installs Python, git, build-essential, corepack,
   uv — identical every single time. Publish a base image once:

  # ghcr.io/theexperiencecompany/gaia-ci-base:latest
  FROM node:22.15.1-bookworm-slim
  RUN apt-get update && apt-get install -y --no-install-recommends \
      python3 python3-pip python3-venv python3-dev git curl build-essential
  libpq-dev \
      && apt-get clean && rm -rf /var/lib/apt/lists/*
  RUN corepack enable && corepack prepare pnpm@10.17.1 --activate
  RUN pip install --break-system-packages uv

  Then ci_env() becomes:
  dag.container()
      .from_("ghcr.io/theexperiencecompany/gaia-ci-base:latest")
      .with_mounted_cache(...)
      .with_directory("/app", source)
      .with_exec(["pnpm", "install", "--frozen-lockfile"])
      .with_exec(["uv", "sync", "--frozen", ...])

  Saves ~30-40s per job. Across 7 parallel jobs that's not huge on wall clock,
  but it's less flaky and cleaner.

  3. Move coverage to a separate non-blocking check

  --cov + --cov-report=term-missing + --cov-fail-under=80 adds overhead to every
   test run. Coverage instrumentation slows pytest down ~10-20%. Run it as a
  separate informational job, or only enforce on master pushes, not every PR.

  With these changes

  detect (30s)
    ├─► test-backend (60s setup + 30s services + 5 min parallel tests) = ~7 min
    ├─► everything else finishes in < 5 min
    └─► quality-gate

  Critical path: ~8 min

  That's a 4-5x improvement from the current 30-40 min monolith.

  The priorities in order of impact:
  1. pytest-xdist — biggest win, easiest change (add one dependency + one flag)
  2. All parallel, no sequential gates — just remove the needs dependencies
  3. Pre-built base image — moderate win, one-time setup
  4. Coverage as separate check — nice to have

  Want me to fold these into the plan and implement?
